 Edit Target Scenario - Split Column Displays Hierarchy or Attribute Codes Instead of Meaningful Descriptions External Bug 2023.4.1.0 Added configuration to support meaningful options for the display of Hierarchy and Attribute codes and descriptions

Notes

New system policy Config: Code/Description Display determines how things generated from a code/description pair will default. Specifically, it is used to generate segment names when Auto-Generate Segments is used; and it is used to display the Segment Split Type information (either hierarchy level or attribute). For more information, refer to the System Policies section.

Target Choice Counts Remove duplicate cluster names within a common cluster set ID External Bug 2023.4.1.0 Reported Behavior

Invalid and duplicated cluster sets with the same Cluster Set ID were showing up Target Choice Count → Segment Definitions, which could possibly result in an error when saving changes.

Notes

The fix for this issue only removes the duplicates of the same Cluster Set with the same Cluster Set ID. However, if there are duplicate names with different Cluster Set ID, it will be listed accordingly in the UI, and will require renaming to distinguish one Cluster Set from another.

Target Choice Counts Update Refresh Rate view displaying incorrect ongoing vs new choices External Bug 2023.4.1.0 Reported Behavior

When using positional merge and cluster constraints, the carryover and new choice counts are incorrect. In some scenarios, the carryover count within a subsequent update period greater than the choices available in the prior update period.

Steps to Verify

  1. Create a target creation scenario

  2. Create a target segment using default options

  3. Use positional merge and cluster constraints

  4. Create multiple update periods

  5. Calculate and rationalize

  6. Target Choices By Update tab

  7. Select all segments and locations

  8. Click the auto-merge function

  9. Review the carryover vs the new choices for each update to confirm it is working

Updated Behavior

Choice counts (carryover and new) are correct after addressing this bug.
Target Choice Counts Decreasing Target Scenario Value causing planned sales to drop significantly below target value External Bug 2023.4.1.0 Reported Behavior

A user creates a target creation scenario including the target segments and update periods. The user then calculates and rationalizes the scenario. At this point the user decides to decrease the target scenario value. This prompts the system to display a message that the scenario needs to be rationalized. The user does this and after doing so finds that the TY planned values do not match the updated scenario target and instead are well below the new value.

Steps to Reproduce

  1. Create a target creation scenario

  2. Create target segments using default options

  3. Create at least 1 update period

  4. Calculate and Rationalize the Scenario

  5. Decrease the Scenario Target by some value (say 25%)

  6. Click Save

  7. System will prompt you to re-actualize

  8. Re-actualize

  9. Review the planned sales values

After making changes to the phase dates of items in the Target Choices By Update tab the expectation is that the Target New Choices tab will reflect updated values based phase date changes.

  2. Add updates

  5. Merge choices or use the auto-merge function to create multiple merges.

  6. Go back to the Target New Choices tab and add in the unique collection metric

  7. Review the total sales or sales units and notice that it is only showing data from the first update
