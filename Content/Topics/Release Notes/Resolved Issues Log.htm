Assortment Seasonality Curve Assignment via Hindsight new item creation incorrect External Bug 2023.4.2.0
Reported Behavior

When creating a new item via the hindsight view the expectation is that the new item modeled after the historic item will inherit the ROS values as well as cluster sets and attributes depending on the selection by the user. The seasonality curve however should be based on the default seasonality curve via the AP plan defaults. The issue we are finding is that the new item is getting a seasonality curve assignment based on a clone of the the cloned seasonality curve of the historic item. The issue with this is that the new items are not capturing the latest calculation of the seasonality curve.

Solution and Modified Behavior

A new system policy, Algorithm To Match Static Curve to Placeholder, determines what mechanism is used to copy a placeholder from like style where the style has a static curve assigned. For more information, refer to the System Policies section.

Components Summary Issue Type Merged to Other Releases
Assortment More consistency in Leadtime (Supplier-DC; DC-Cluster) calculations/applications External Bug 2023.4.2.0
Reported Behavior

The work performed for this issue addresses an inconsistency in the way lead time is calculated depending on its use.

The Assortment Receipt Plan Initial Select and Assortment Flow Dates Generation process both have a similar 'nearest' date logic but…

  • the RPV would only consider the default lead-time system policy.

  • The flow dates process only uses AP Defaults first then system policy.

Both of those differ from the engine which uses the placeholder's variant’s default supplier's default lead time then attribute-based defaults then system policy.

This discrepancy can cause, among other things, the editable range in the RPV to differ from the system generated flow dates. For some customers, they could be seeing the RPV editable periods as 28 days(system policy) in the future and the flow dates beginning 16 days (supplier lead-time) in the future. So receipt units are being generated in periods that a user cannot maintain.

Solution and Updated Behavior

New system policy:

Name: Assortment Engine Supplier-to-DC Lead Time

Renamed system policy:

Old name: Assortment Engine DC-to-Store Lead Time

New name: Assortment Engine DC-to-Cluster Lead Time

For descriptions, see the System Policies section.

The earliest planned flow date will now be greater than or equal to the earliest editable period in the RPV.

Components Summary Issue Type Merged to Other Releases
Import/Export SQL API - Import of Supplier Order Line Variant generating Warnings If UploadLineUUID Not Populated External Bug  
Background

Supplier Orders including Supplier Order Line Variants, can come from a number of external sources for a customer and may not have originated from a JustEnough generated commitment or distribution order.

Therefore the UploadLineUUID can be null (i.e. the supplier order does not link to an order originally generated within JustEnough).

The data spec for supplier order line variant correctly identifies the UploadLineUUID as preferred but not required.

Unfortunately the Supplier Order line Variant import validation generates a warning for the records if the UploadLineUUID is not populated (NULL) causing a lot of unwanted ‘noise’ in the import messages.

In contrast, the Supplier Order Line import does not generate warnings if the UploadLineUUID is not populated which is the expected behavior for a field that may not be populated for a good reason.

Reported Behavior

Importing a supplier order line variant record with the UploadLineUUID not populated, would generate a warning to say the UploadLineUUID is missing.

Modified Behavior

Importing a supplier order line variant record without the UploadLineUUID being populated no longer generates the warning
