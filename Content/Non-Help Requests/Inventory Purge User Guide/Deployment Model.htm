Deployment Model

The core code deployment includes executing the purge-specific INIT script. By design, the initial setup deploys reference data, then adds the prescription to follow to effect Causal purges.

When the solution is refreshed by a hot-fix or Service Pack, the same INIT script is run. The script does what it can to preserve Custom prescription steps. The only time such steps are removed is when we remove a core-code step-type. By definition, removal of a step-type includes removal of any logic performed when that step-type existed in prior versions.

The purge utility is not versioned separately from the overall solution version.

We recommend that the Custom INIT script be kept under version control. Any custom step that runs custom procedural code would recommend that the custom code also be managed under version control.

The utility is provided in a form whereby successful execution of core code is the norm. We are not responsible for any custom steps or code that act upon custom tables, or that act upon core tables but not within core code. Complete and thorough testing of custom facets is required in a non-production environment prior to production usage. Data recovery of information losses due to custom artifacts is the responsibility of the customer where we have not been engaged to provide said custom artifacts in a thoroughly tested form.

It is incumbent upon Support, for hosted implementations, or the on-premise Customer, to review Job Manager Tasks to ensure that the latter procedure is not directly referenced. If it is, it must be removed. It is recommended to only use Method names in Job Manager.

Ensure that the @UserID value is assigned to the currently logged-in User ID, or accept that defaulting to UserID = 1 reduces traceability in regards to the person explicitly starting the Task. For the Stop Now Task, ensure that the parameter value is True.
